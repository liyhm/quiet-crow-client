# 消息不显示问题 - 后端排查指南

## 🔍 问题描述

前端重启应用后，左侧会话列表显示"共3条消息"，但调用历史消息接口返回空数组，导致右侧消息列表为空。

## 📊 前端调用情况

### 1. 获取会话列表接口

**接口**: `GET /api/session/my`

**请求头**:
```
Authorization: Bearer {JWT_TOKEN}
```

**前端期望响应**:
```json
[
  {
    "id": "session-uuid-123",
    "sessionId": "session-uuid-123",
    "userId": "user-uuid-456",
    "showName": "lyh1",
    "avatar": null,
    "sessionUserType": "PRIVATE",
    "lastMessage": "最后一条消息内容",
    "lastMessageTime": "2024-02-27T12:00:00",
    "unreadCount": 3
  }
]
```

**问题**: 这个接口返回正常，显示有 3 条未读消息

---

### 2. 获取历史消息接口（问题接口）

**接口**: `GET /api/session/{sessionId}/messages`

**请求参数**:
- `sessionId`: 会话ID（从会话列表获取）
- `cursor`: 可选，分页游标（前端传 `undefined`）
- `limit`: 消息数量限制（前端传 `50`）

**完整请求示例**:
```

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

**前端期望响应**:

## 🎯 后端需要排查的问题

### 问题 1: 数据库中是否有消息记录？

**SQL 查询**:
```sql
-- 查询指定会话的所有消息
SELECT * FROM message 
WHERE session_id = '1274342b142fc61dd1806425b9c0ce30'
ORDER BY send_time DESC;

-- 查询消息数量
SELECT COUNT(*) FROM message 
WHERE session_id = '1274342b142fc61dd1806425b9c0ce30';
```

**预期结果**: 应该有 3 条消息记录

**如果没有记录**: 
- 检查消息是否被物理删除
- 检查 session_id 是否正确
- 检查消息保存逻辑是否有问题

---

### 问题 2: 是否有水位线过滤？

**可能的过滤逻辑**:
```java
// 检查是否有类似这样的代码
public List<Message> getMessages(String sessionId, String userId) {
    // 获取用户的水位线
    SessionUser sessionUser = sessionUserRepository
        .findBySessionIdAndUserId(sessionId, userId);
    
    Long watermark = sessionUser.getWatermark();
    
    // 只返回水位线之后的消息
    return messageRepository
        .findBySessionIdAndIdGreaterThan(sessionId, watermark);
}
```

**问题**: 如果用户的水位线设置在最后一条消息之后，会导致返回空数组

**排查方法**:
```sql
-- 查询用户的水位线
SELECT * FROM session_user 
WHERE session_id = '1274342b142fc61dd1806425b9c0ce30'
AND user_id = '当前用户ID';

-- 查看 watermark 字段的值
-- 如果 watermark 大于等于最后一条消息的 ID，就会过滤掉所有消息
```

---

### 问题 5: 是否有分页参数问题？

**可能的问题**:
```java
// 如果 cursor 参数处理不当
public List<Message> getMessages(String sessionId, String cursor, int limit) {
    if (cursor != null && cursor.equals("undefined")) {
        // 错误：把字符串 "undefined" 当作有效游标
```

**后端接收**:

**排查方法**: 在后端接口打印参数

## 🔧 后端排查步骤

### 步骤 1: 添加详细日志

在 `getMessages` 接口中添加日志：

```java
@GetMapping("/{sessionId}/messages")
public ResponseEntity<List<MessageDTO>> getMessages(
    @PathVariable String sessionId,
    @RequestParam(required = false) String cursor,
    @RequestParam(defaultValue = "50") int limit,
    @AuthenticationPrincipal UserDetails userDetails
) {
    String userId = userDetails.getUsername();
    
    log.info("========================================");
    log.info("[getMessages] 接收到请求");
    log.info("[getMessages] sessionId: {}", sessionId);
    log.info("[getMessages] cursor: {}", cursor);
    log.info("[getMessages] limit: {}", limit);
    log.info("[getMessages] userId: {}", userId);
    
    // 1. 检查会话是否存在
    Session session = sessionRepository.findById(sessionId).orElse(null);
    log.info("[getMessages] 会话是否存在: {}", session != null);
    
    // 2. 检查用户是否是会话成员
    SessionUser sessionUser = sessionUserRepository
        .findBySessionIdAndUserId(sessionId, userId)
        .orElse(null);
    log.info("[getMessages] 用户是否是会话成员: {}", sessionUser != null);
    
    if (sessionUser != null) {
        log.info("[getMessages] 用户水位线: {}", sessionUser.getWatermark());
    }
    
    // 3. 查询数据库中的消息
    List<Message> allMessages = messageRepository.findBySessionId(sessionId);
    log.info("[getMessages] 数据库中的消息总数: {}", allMessages.size());
    
    // 4. 应用过滤逻辑
    List<Message> filteredMessages = applyFilters(allMessages, userId, sessionUser);
    log.info("[getMessages] 过滤后的消息数: {}", filteredMessages.size());
    
    // 5. 转换为 DTO
    List<MessageDTO> result = filteredMessages.stream()
        .map(this::toDTO)
        .collect(Collectors.toList());
    
    log.info("[getMessages] 最终返回消息数: {}", result.size());
    log.info("========================================");
    
    return ResponseEntity.ok(result);
}
```

### 步骤 2: 检查数据库

```sql
-- 1. 检查会话是否存在
SELECT * FROM session WHERE id = '1274342b142fc61dd1806425b9c0ce30';

-- 2. 检查会话成员
SELECT * FROM session_user 
WHERE session_id = '1274342b142fc61dd1806425b9c0ce30';

SELECT id, send_user, message_content, send_time, is_read, deleted_by

SELECT COUNT(*) as total FROM message

```

### 步骤 3: 检查过滤逻辑

找到消息查询的代码，检查是否有以下过滤：

1. **水位线过滤**
   ```java
   // 查找类似代码
   .andIdGreaterThan(watermark)
   .andSendTimeAfter(watermarkTime)
   ```

2. **软删除过滤**
   ```java
   // 查找类似代码
   .andDeletedByNotContaining(userId)
   .andIsDeletedFalse()
   ```

3. **权限过滤**
   ```java
   // 查找类似代码
   if (!isSessionMember(sessionId, userId)) {
       return Collections.emptyList();
   }
   ```

### 步骤 4: 临时禁用过滤

为了快速定位问题，可以临时注释掉所有过滤逻辑：

```java
@GetMapping("/{sessionId}/messages")
public ResponseEntity<List<MessageDTO>> getMessages(
    @PathVariable String sessionId,
    @RequestParam(required = false) String cursor,
    @RequestParam(defaultValue = "50") int limit
) {
    // 临时：直接返回所有消息，不做任何过滤
    List<Message> messages = messageRepository.findBySessionId(sessionId);
    
    log.info("临时测试 - 返回所有消息，数量: {}", messages.size());
    
    List<MessageDTO> result = messages.stream()
        .map(this::toDTO)
        .collect(Collectors.toList());
    
    return ResponseEntity.ok(result);
}
```

如果这样能返回消息，说明问题在过滤逻辑中。

## 🎯 最可能的原因

根据经验，最可能的原因是：

### 1. 水位线过滤（80%）

用户的水位线被设置在最后一条消息之后，导致所有消息被过滤。

**解决方法**:
- 检查水位线更新逻辑
- 重置用户的水位线为 0
- 或者修改查询逻辑，不使用水位线过滤历史消息

### 2. 软删除过滤（15%）

消息被标记为已删除（对当前用户）。

**解决方法**:
- 检查删除逻辑
- 清除 deleted_by 字段
- 或者修改查询逻辑

### 3. 数据库中确实没有消息（5%）

消息没有保存成功或被物理删除。

**解决方法**:
- 检查消息保存逻辑
- 检查数据库事务
- 恢复数据
