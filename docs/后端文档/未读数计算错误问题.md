# 未读数计算错误问题

## 🔍 问题描述

会话列表中显示的未读数不准确。例如：
- 实际只有 1 条未读消息
- 但显示为 2 条未读

## 📊 问题截图分析

从用户截图看到：
- 左侧会话列表显示：`lyh1` 会话有 `2` 个未读（红色圆圈）
- 但实际可能只有 1 条新消息

## 🎯 涉及的接口

### 获取会话列表接口

**接口**: `GET /api/session/my`

**前端期望响应**:
```json
[
  {
    "id": "session-uuid-123",
    "sessionId": "session-uuid-123",
    "userId": "user-uuid-456",
    "showName": "lyh1",
    "avatar": null,
    "sessionUserType": "PRIVATE",
    "lastMessage": "最后一条消息内容（加密）",
    "lastMessageTime": "2024-02-27T12:00:00",
    "unreadCount": 2  ← 这个值不准确
  }
]
```

## 🔧 可能的原因

### 原因 1: 未读数计算逻辑错误（最可能）

**问题代码示例**:
```java
// 错误的计算方式
public int calculateUnreadCount(String sessionId, String userId) {
    // 获取用户的水位线
    SessionUser sessionUser = sessionUserRepository
        .findBySessionIdAndUserId(sessionId, userId)
        .orElse(null);
    
    if (sessionUser == null) {
        return 0;
    }
    
    Long watermark = sessionUser.getWatermark();
    
    // 错误：计算所有消息，而不是只计算对方发送的
    return messageRepository.countBySessionIdAndIdGreaterThan(sessionId, watermark);
}
```

**正确的计算方式**:
```java
public int calculateUnreadCount(String sessionId, String userId) {
    SessionUser sessionUser = sessionUserRepository
        .findBySessionIdAndUserId(sessionId, userId)
        .orElse(null);
    
    if (sessionUser == null) {
        return 0;
    }
    
    Long watermark = sessionUser.getWatermark();
    
    // 正确：只计算对方发送的、在水位线之后的消息
    return messageRepository.countBySessionIdAndIdGreaterThanAndSendUserNot(
        sessionId, 
        watermark, 
        userId  // 排除自己发送的消息
    );
}
```

### 原因 2: 水位线更新时机错误

**问题场景**:
1. 用户 A 发送消息
2. 后端保存消息，消息 ID = 100
3. 后端错误地更新了用户 A 的水位线为 99（应该是 100）
4. 导致用户 A 自己的消息也被计入未读数

**正确的逻辑**:
```java
// 发送消息时
public void sendMessage(String sessionId, String senderId, String content) {
    // 1. 保存消息
    Message message = new Message();
    message.setSessionId(sessionId);
    message.setSendUser(senderId);
    message.setMessageContent(content);
    message = messageRepository.save(message);
    
    // 2. 更新发送者的水位线（标记为已读）
    SessionUser senderSessionUser = sessionUserRepository
        .findBySessionIdAndUserId(sessionId, senderId)
        .orElseThrow();
    
    senderSessionUser.setWatermark(message.getId());  // 发送者立即标记为已读
    sessionUserRepository.save(senderSessionUser);
    
    // 3. 不更新接收者的水位线（保持未读状态）
}
```

### 原因 3: 标记已读时未正确更新水位线

**问题场景**:
1. 用户打开会话，前端调用标记已读接口
2. 后端更新水位线时出错
3. 导致未读数没有清零

**检查代码**:
```java
@PostMapping("/session/{sessionId}/read/batch")
public void markBatchAsRead(
    @PathVariable String sessionId,
    @RequestBody List<String> messageIds,
    @AuthenticationPrincipal UserDetails userDetails
) {
    String userId = userDetails.getUsername();
    
    // 获取最大的消息 ID
    Long maxMessageId = messageIds.stream()
        .map(id -> messageRepository.findById(id).orElse(null))
        .filter(Objects::nonNull)
        .map(Message::getId)
        .max(Long::compareTo)
        .orElse(null);
    
    if (maxMessageId != null) {
        // 更新水位线
        SessionUser sessionUser = sessionUserRepository
            .findBySessionIdAndUserId(sessionId, userId)
            .orElseThrow();
        
        sessionUser.setWatermark(maxMessageId);
        sessionUserRepository.save(sessionUser);
        
        log.info("更新水位线 - sessionId: {}, userId: {}, watermark: {}", 
            sessionId, userId, maxMessageId);
    }
}
```

### 原因 4: 并发问题

**问题场景**:
1. 用户快速发送多条消息
2. 多个请求并发更新水位线
3. 导致水位线值不一致

**解决方法**:
```java
// 使用数据库锁
@Transactional
public void updateWatermark(String sessionId, String userId, Long messageId) {
    SessionUser sessionUser = sessionUserRepository
        .findBySessionIdAndUserIdForUpdate(sessionId, userId)  // 加锁
        .orElseThrow();
    
    // 只有当新的消息 ID 更大时才更新
    if (messageId > sessionUser.getWatermark()) {
        sessionUser.setWatermark(messageId);
        sessionUserRepository.save(sessionUser);
    }
}
```

## 📝 排查步骤

### 步骤 1: 检查数据库

```sql
-- 1. 查询会话成员信息
SELECT 
    su.session_id,
    su.user_id,
    su.watermark,
    u.username
FROM session_user su
LEFT JOIN user u ON su.user_id = u.id
WHERE su.session_id = '你的会话ID';

-- 2. 查询消息记录
SELECT 
    id,
    send_user,
    message_content,
    send_time,
    is_read
FROM message
WHERE session_id = '你的会话ID'
ORDER BY id DESC
LIMIT 10;

-- 3. 手动计算未读数
-- 对于用户 A：
SELECT COUNT(*) as unread_count
FROM message
WHERE session_id = '你的会话ID'
AND id > (
    SELECT watermark 
    FROM session_user 
    WHERE session_id = '你的会话ID' 
    AND user_id = '用户A的ID'
)
AND send_user != '用户A的ID';  -- 排除自己发送的消息
```

### 步骤 2: 添加日志

在 `getMySessions` 接口中添加详细日志：

```java
@GetMapping("/my")
public List<SessionUserDTO> getMySessions(@AuthenticationPrincipal UserDetails userDetails) {
    String userId = userDetails.getUsername();
    
    log.info("========================================");
    log.info("[getMySessions] 获取会话列表");
    log.info("[getMySessions] userId: {}", userId);
    
    List<SessionUser> sessionUsers = sessionUserRepository.findByUserId(userId);
    log.info("[getMySessions] 会话数量: {}", sessionUsers.size());
    
    List<SessionUserDTO> result = new ArrayList<>();
    
    for (SessionUser su : sessionUsers) {
        log.info("----------------------------------------");
        log.info("[getMySessions] 处理会话: {}", su.getSessionId());
        log.info("[getMySessions] 用户水位线: {}", su.getWatermark());
        
        // 计算未读数
        int unreadCount = calculateUnreadCount(su.getSessionId(), userId);
        log.info("[getMySessions] 计算的未读数: {}", unreadCount);
        
        // 手动验证
        long totalMessages = messageRepository.countBySessionId(su.getSessionId());
        long messagesAfterWatermark = messageRepository
            .countBySessionIdAndIdGreaterThan(su.getSessionId(), su.getWatermark());
        long myMessages = messageRepository
            .countBySessionIdAndSendUser(su.getSessionId(), userId);
        
        log.info("[getMySessions] 总消息数: {}", totalMessages);
        log.info("[getMySessions] 水位线之后的消息数: {}", messagesAfterWatermark);
        log.info("[getMySessions] 我发送的消息数: {}", myMessages);
        log.info("[getMySessions] 预期未读数: {}", messagesAfterWatermark - myMessages);
        
        SessionUserDTO dto = new SessionUserDTO();
        dto.setSessionId(su.getSessionId());
        dto.setUnreadCount(unreadCount);
        // ... 其他字段
        
        result.add(dto);
    }
    
    log.info("========================================");
    return result;
}
```

## 🎯 预期的正确行为

### 发送消息时

| 操作 | 发送者未读数 | 接收者未读数 |
|------|------------|------------|
| 用户 A 发送消息 | 0（自己的消息不算未读） | +1 |
| 用户 B 发送消息 | +1 | 0（自己的消息不算未读） |

### 标记已读时

| 操作 | 未读数变化 |
|------|----------|
| 用户 A 打开会话 | 用户 A 的未读数清零 |
| 用户 B 打开会话 | 用户 B 的未读数清零 |

### 水位线更新规则

1. **发送消息时**: 发送者的水位线立即更新到该消息 ID
2. **接收消息时**: 接收者的水位线不变（保持未读）
3. **打开会话时**: 前端调用标记已读接口，更新水位线
4. **并发情况**: 水位线只能增加，不能减少
