# 后端需求 - 消息已读状态优化

## 需求背景

**当前问题**：
- 每次刷新页面，所有消息都显示为未读
- 会话列表的未读数不准确
- 用户已读的消息，刷新后又变成未读

**期望效果**：
- 用户标记消息已读后，状态持久化保存
- 刷新页面后，已读状态保持不变
- 会话列表正确显示未读消息数量

## 核心需求

### 需求 1：优化标记消息已读接口

**接口**：`POST /api/session/{sessionId}/read`

**当前实现问题**：
- 可能没有正确更新数据库
- 或者更新逻辑有问题

**功能描述**：
标记指定消息及之前的所有消息为已读。

**请求参数**：
```json
{
  "messageId": "message-uuid"
}
```

**处理逻辑**：
1. 获取当前用户ID
2. 查询指定消息的发送时间
3. 更新 `message` 表：
   - 条件：会话ID匹配 AND 发送时间 <= 指定消息时间 AND 发送者不是当前用户
   - 操作：设置 `is_read = true`
4. 返回成功响应

**注意事项**：
- 只标记对方发送的消息（不标记自己发送的）
- 使用时间范围批量更新（提高性能）
- 确保数据库事务正确提交

### 需求 2：获取历史消息时返回正确的已读状态

**接口**：`GET /api/session/{sessionId}/messages`

**功能描述**：
返回消息列表时，每条消息的 `isRead` 字段应该反映真实的已读状态。

**查询逻辑**：
- 从 `message` 表查询消息
- 返回 `is_read` 字段的真实值
- 不要所有消息都返回 `false`

**返回格式**：
```json
{
  "code": 200,
  "data": [
    {
      "id": "message-uuid",
      "sessionId": "session-uuid",
      "sendUser": "user-uuid",
      "messageContent": "加密内容",
      "messageType": "1",
      "sendTime": "2024-02-27T10:00:00",
      "isRead": true  // 真实的已读状态
    }
  ]
}
```

### 需求 3：获取会话列表时返回未读消息数

**接口**：`GET /api/session/my`

**功能描述**：
返回会话列表时，包含每个会话的未读消息数量。

**计算逻辑**：
1. 对于每个会话
2. 查询该会话中：
   - 发送者不是当前用户
   - `is_read = false`
   - 的消息数量
3. 返回未读数

**返回格式**：
```json
{
  "code": 200,
  "data": [
    {
      "id": "session-user-uuid",
      "sessionId": "session-uuid",
      "userId": "friend-user-uuid",
      "showName": "张三",
      "avatar": "http://...",
      "sessionUserType": "PRIVATE",
      "lastMessage": "你好",
      "lastMessageTime": "2024-02-27T14:30:00",
      "unreadCount": 3  // 未读消息数量
    }
  ]
}
```

## 数据库查询参考

### 标记消息已读

**需要更新的记录**：
- 表：`message`
- 条件：
  - `session_id = ?` （会话ID）
  - `send_user != ?` （不是当前用户发送的）
  - `send_time <= ?` （发送时间早于或等于指定消息）
  - `is_read = false` （当前未读）
- 操作：
  - 设置 `is_read = true`

### 查询未读消息数

**查询条件**：
- 表：`message`
- 条件：
  - `session_id = ?` （会话ID）
  - `send_user != ?` （不是当前用户发送的）
  - `is_read = false` （未读）
- 操作：
  - COUNT(*) 统计数量

## 测试验证

### 验证步骤 1：标记已读

1. 用户A发送3条消息给用户B
2. 用户B打开会话，查看消息
3. 前端调用 `POST /api/session/{sessionId}/read`
4. 刷新页面
5. 验证：消息仍然显示为已读，未读数为0

### 验证步骤 2：未读数统计

1. 用户A发送5条消息给用户B
2. 用户B未打开会话
3. 调用 `GET /api/session/my`
4. 验证：该会话的 `unreadCount` 为 5
5. 用户B打开会话并标记已读
6. 再次调用 `GET /api/session/my`
7. 验证：该会话的 `unreadCount` 为 0

### 验证步骤 3：部分已读

1. 用户A发送10条消息
2. 用户B打开会话，滚动到第5条
3. 前端标记第5条消息已读
4. 验证：前5条消息 `is_read = true`，后5条 `is_read = false`
5. 未读数应该是 5

## 常见问题

### 问题 1：标记已读后，数据库没有更新

**可能原因**：
- 事务没有提交
- 更新条件不正确
- 权限问题

**排查方法**：
- 查看数据库日志
- 检查更新语句是否执行
- 验证更新条件是否匹配到记录

### 问题 2：未读数不准确

**可能原因**：
- 查询条件错误（包含了自己发送的消息）
- 没有过滤已读消息
- 统计逻辑有问题

**排查方法**：
- 打印查询SQL
- 手动执行SQL验证结果
- 检查查询条件

### 问题 3：刷新后已读状态丢失

**可能原因**：
- 标记已读接口没有真正更新数据库
- 查询消息时没有返回真实的 `is_read` 值
- 前端缓存问题

**排查方法**：
- 直接查询数据库验证 `is_read` 字段
- 检查接口返回的数据
- 清除前端缓存测试

## 补充说明

### 关于性能优化

- 标记已读使用批量更新（按时间范围）
- 未读数统计可以考虑缓存
- 避免每次都全表扫描

### 关于并发问题

- 多个设备同时标记已读时，以最新的为准
- 使用数据库的原子操作避免冲突

### 关于消息推送

- 当消息被标记已读时，可以通过 WebSocket 通知发送者
- 显示"已读"状态（类似微信的双勾）
- 这个功能可以后续实现
