## 1. 🔴 修复联系人列表返回数据

### 接口
`GET /api/contact/list`

### 问题
返回的数据中 `userId` 字段为 `null` 或缺失

### 需求
确保返回的每个联系人对象包含以下字段：
- `userId` - 好友的用户ID（必需，不能为 null）
- `username` - 好友的用户名
- `showName` - 好友的昵称
- `avatar` - 好友的头像URL
- `remark` - 备注（可为 null）
- `addedTime` - 添加时间
- `publicKey` - 公钥

### 注意事项
- `userId` 必须是好友的用户ID，不是关系ID，也不是当前用户的ID
- 好友关系是双向的，需要判断当前用户是 fromUserId 还是 toUserId
- 必须关联用户表获取好友的完整信息

### 验证方法
返回的数据格式应该是：
```json
{
  "code": 200,
  "data": [
    {
      "id": "relation-uuid",
      "userId": "friend-user-uuid",  // 必需，不能为 null
      "username": "zhangsan",
      "showName": "张三",
      "avatar": "http://...",
      "remark": null,
      "addedTime": "2024-02-27T10:00:00",
      "publicKey": "..."
    }
  ]
}
```

## 2. 🔴 修复会话列表返回数据

### 接口
`GET /api/session/my`

### 问题
返回的数据中 `showName` 字段为 `null`

### 需求
确保返回的每个会话对象包含以下字段：
- `sessionId` - 会话ID（必需）
- `userId` - 对方用户ID（必需）
- `showName` - 对方用户昵称或群名称（必需，不能为 null）
- `sessionUserType` - 会话类型（PRIVATE 或 GROUP）

### 业务逻辑
- **私聊会话**: `showName` 应该是对方用户的昵称（从用户表查询）
- **群聊会话**: `showName` 应该是群名称（从会话表查询）
- 如果查询不到昵称，至少返回用户名，不要返回 null

### 验证方法
返回的数据格式应该是：
```json
{
  "code": 200,
  "data": [
    {
      "id": "session-user-uuid",
      "sessionId": "session-uuid",
      "userId": "friend-user-uuid",
      "showName": "张三",  // 必需，不能为 null
      "sessionUserType": "PRIVATE"
    }
  ]
}
```

## 3. 🔴 实现创建私聊会话

### 接口
`POST /api/session/private`

### 请求参数
```json
{
  "targetUserId": "user-uuid"
}
```

### 需求
1. 检查当前用户和目标用户之间是否已存在私聊会话
2. 如果已存在，返回现有会话ID（`exists: true`）
3. 如果不存在，创建新会话：
   - 在 `session` 表创建会话记录（type: PRIVATE）
   - 在 `session_user` 表为两个用户都创建记录
   - 返回新会话ID（`exists: false`）

### 响应格式
```json
{
  "code": 200,
  "data": {
    "sessionId": "session-uuid",
    "type": "PRIVATE",
    "participants": ["current-user-id", "target-user-id"],
    "exists": false
  }
}
```

### 业务规则
- 一对用户之间只能有一个私聊会话
- 需要验证目标用户是否存在
- 需要验证两个用户是否是好友关系（可选）

## 4. 🟡 实现删除会话（软删除）

### 接口
`DELETE /api/session/{sessionId}`

### 需求
删除当前用户的会话记录（软删除）

### 业务逻辑
- 只删除当前用户的 `session_user` 记录
- 不删除 `session` 表记录
- 不删除 `message` 表记录
- 不影响其他用户（对方仍能看到会话）

### 验证
- 验证用户是否有权限删除该会话
- 验证会话是否存在

### 响应格式
```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

## 5. 🟡 实现获取历史消息

### 接口
`GET /api/session/{sessionId}/messages`

### 请求参数
- `cursor` - 游标（可选，用于分页）
- `limit` - 每页数量（默认50）

### 需求
返回指定会话的历史消息列表

### 响应格式
```json
{
  "code": 200,
  "data": [
    {
      "id": "message-uuid",
      "sessionId": "session-uuid",
      "sendUser": "user-uuid",
      "messageContent": "加密的消息内容",
      "messageType": "TEXT",
      "sendTime": "2024-02-27T10:00:00",
      "isRead": false
    }
  ]
}
```

### 业务规则
- 验证用户是否是会话成员
- 按时间倒序返回（最新的在前）
- 支持分页加载

## 6. 🟡 实现 WebSocket 消息转发

### 功能
接收客户端发送的消息，保存到数据库，并转发给会话中的其他用户

### 消息格式
**接收**:
```json
{
  "type": "CHAT_MESSAGE",
  "sessionId": "session-uuid",
  "content": "加密的消息内容",
  "messageType": "TEXT"
}
```

**转发**:
```json
{
  "type": "CHAT_MESSAGE",
  "messageId": "message-uuid",
  "sessionId": "session-uuid",
  "sendUser": "sender-user-uuid",
  "content": "加密的消息内容",
  "messageType": "TEXT",
  "sendTime": "2024-02-27T10:00:00"
}
```

### 业务逻辑
1. 验证用户是否是会话成员
2. 保存消息到 `message` 表
3. 查找会话中的其他在线用户
4. 通过 WebSocket 转发消息给其他用户
5. 更新会话的最后消息时间

## 数据库表结构参考

### session 表
- `id` - 会话ID（主键）
- `session_type` - 会话类型（PRIVATE, GROUP）
- `group_name` - 群名称（群聊时使用）
- `created_time` - 创建时间
- `updated_time` - 更新时间

### session_user 表
- `id` - 记录ID（主键）
- `session_id` - 会话ID（外键）
- `user_id` - 用户ID（外键）
- `session_user_type` - 类型（PRIVATE, GROUP）
- `join_time` - 加入时间

### message 表
- `id` - 消息ID（主键）
- `session_id` - 会话ID（外键）
- `send_user` - 发送者ID（外键）
- `message_content` - 消息内容（加密）
- `message_type` - 消息类型（TEXT, IMAGE, FILE）
- `send_time` - 发送时间
- `is_read` - 是否已读

### contact_relation 表
- `id` - 关系ID（主键）
- `from_user_id` - 发起用户ID
- `to_user_id` - 目标用户ID
- `status` - 状态（PENDING, ACCEPTED, REJECTED）
- `remark` - 备注
- `added_time` - 添加时间
