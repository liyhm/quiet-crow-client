# 后端需求 - 首条消息接收问题修复

## 问题描述

**现象**:
- 用户 A 给用户 B 发送第一条消息
- 用户 B 的接收端没有收到消息
- 用户 A 发送第二条消息
- 用户 B 能收到第二条消息（以及后续所有消息）

**根本原因**:
1. 第一条消息发送时，接收方（用户 B）的 `b_session_user` 表中没有记录
2. 前端 WebSocket 收到消息后，检测到会话不存在
3. 前端调用 `loadSessions()` 重新加载会话列表
4. 但此时消息已经处理完了，第一条消息丢失
5. 第二条消息发送时，会话已经存在，能正常接收

---

## 解决方案（推荐）

### 后端发送消息时自动创建接收方的 session_user

**核心逻辑**:
1. 在 `handleChatMessage()` 方法中，保存消息后
2. 获取会话的所有成员列表
3. 遍历每个成员，检查是否有 `b_session_user` 记录
4. 如果没有记录，自动创建（`is_visible = true`, `clear_watermark = 0`）
5. 如果有记录但 `is_visible = false`（被删除了），恢复为 `true`
6. 然后再转发消息给所有成员

**实现位置**: 
- 文件: `ChatMessageService.java`
- 方法: `handleChatMessage()`
- 新增方法: `ensureSessionUserExists(String sessionId, String userId)`

**业务流程**:
```
收到消息
    ↓
保存到 b_session_log
    ↓
获取会话成员列表
    ↓
遍历每个成员：
    检查 b_session_user 是否存在
        不存在 → 创建新记录（is_visible=true, clear_watermark=0）
        存在但 is_visible=false → 更新为 true
        存在且 is_visible=true → 跳过
    ↓
转发消息给所有成员（除了发送者）
```

**优点**:
- 前端无需修改
- 彻底解决首条消息丢失问题
- 符合微信的行为（对方发消息，会话自动出现）
- 支持会话复活（删除后收到新消息，会话重新出现）

---

## 需要实现的方法

### ensureSessionUserExists(String sessionId, String userId)

**功能**: 确保指定用户在指定会话中有 `b_session_user` 记录

**逻辑**:
1. 查询 `b_session_user` 表，条件：`session_id = ? AND user_id = ?`
2. 如果记录不存在：
   - 创建新记录
   - 设置 `is_visible = true`
   - 设置 `clear_watermark = 0`
   - 记录日志："自动创建 session_user"
3. 如果记录存在但 `is_visible = false`：
   - 更新 `is_visible = true`
   - 更新 `update_time`
   - 记录日志："恢复 session_user 可见性"
4. 如果记录存在且 `is_visible = true`：
   - 不做任何操作

## 测试验证

### 测试场景 1: 首次发送消息

**步骤**:
1. 用户 A 和用户 B 都登录
2. 用户 A 给用户 B 发送第一条消息
3. 检查用户 B 是否收到消息

**预期结果**:
- ✅ 用户 B 实时收到第一条消息
- ✅ 用户 B 的会话列表中出现该会话
- ✅ 会话显示在列表顶部

**数据库验证**:
```sql
-- 发送消息前
SELECT * FROM b_session_user WHERE session_id = 'xxx';
-- 结果：只有 userA 的记录

-- 发送消息后
SELECT * FROM b_session_user WHERE session_id = 'xxx';
-- 结果：有 userA 和 userB 的记录，userB 的 is_visible = true
```

---

### 测试场景 2: 删除会话后收到消息

**步骤**:
1. 用户 B 删除与用户 A 的会话
2. 用户 A 发送新消息
3. 检查用户 B 是否收到消息

**预期结果**:
- ✅ 用户 B 实时收到消息
- ✅ 会话重新出现在列表顶部
- ✅ 只显示新消息（旧消息被水位线过滤）

**数据库验证**:
```sql
-- 删除会话后
SELECT * FROM b_session_user WHERE session_id = 'xxx' AND user_id = 'userB';
-- 结果：is_visible = false

-- 收到新消息后
SELECT * FROM b_session_user WHERE session_id = 'xxx' AND user_id = 'userB';
-- 结果：is_visible = true（已恢复）
```

## 数据库变化

### 修改前

**b_session_user 表**:
```
用户 A 发送第一条消息前：
只有用户 A 的记录，用户 B 没有记录
```

**问题**: 用户 B 没有 session_user 记录，所以收不到消息

---

### 修改后

**b_session_user 表**:
```
用户 A 发送第一条消息后：
自动创建了用户 B 的记录（is_visible = true, clear_watermark = 0）
```

**效果**: 用户 B 有了 session_user 记录，能收到消息

---

## 注意事项

### 1. 群聊场景

对于群聊，也需要确保所有成员都有 session_user 记录。

建议逻辑：
- 获取会话类型（私聊/群聊）
- 如果是群聊，获取所有群成员列表
- 遍历所有成员，调用 `ensureSessionUserExists()`

### 2. 性能优化

如果担心每次发送消息都查询数据库影响性能，可以考虑：
- 使用 Redis 缓存 session_user 的存在性
- 缓存 key 格式：`session_user:{sessionId}:{userId}`
- 缓存过期时间：1 小时
- 先检查缓存，缓存未命中再查询数据库

## 总结

**问题根源**:
- 接收方没有 session_user 记录
- 前端收到消息时会话不存在
- 第一条消息丢失

**解决方案**:
- 后端发送消息时，自动创建/恢复接收方的 session_user 记录
- 确保接收方的会话列表中有该会话
- 前端收到消息时，会话已存在，能正常处理

**修改位置**:
- 后端 `ChatMessageService.java`
- 添加 `ensureSessionUserExists()` 方法
- 在 `handleChatMessage()` 中调用

**测试验证**:
- 首次发送消息 → 接收方能收到
- 删除会话后收到消息 → 会话自动恢复
