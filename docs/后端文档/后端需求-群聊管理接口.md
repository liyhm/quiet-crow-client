# 后端需求：群聊管理接口

## 📋 需求概述

前端需要两个群聊管理接口：
1. 邀请成员加入群聊
2. 解散群聊

---

## 🔌 接口定义

### 1. 邀请成员加入群聊

**接口路径**: `POST /api/session/{sessionId}/invite`

**请求参数**:
- `sessionId` (路径参数): 群聊的 session ID

**请求头**:
```
Authorization: Bearer {token}
Content-Type: application/json
```

**请求体**:
```json
{
  "memberIds": ["user123", "user456", "user789"]
}
```

**响应格式**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "sessionId": "group-177226863258",
    "invitedCount": 3,
    "failedUsers": []
  }
}
```

**字段说明**:
- `sessionId`: 群聊ID
- `invitedCount`: 成功邀请的人数
- `failedUsers`: 邀请失败的用户列表（可选，如果有用户已在群内或不存在）

---

### 2. 解散群聊

**接口路径**: `DELETE /api/session/{sessionId}/dissolve`

**请求参数**:
- `sessionId` (路径参数): 群聊的 session ID

**请求头**:
```
Authorization: Bearer {token}
```

**响应格式**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "sessionId": "group-177226863258",
    "dissolvedAt": "2024-02-28T10:30:00Z"
  }
}
```

## 🗄️ 数据库操作逻辑

### 邀请成员接口

#### 1. 权限验证
```sql
-- 验证会话是否是群聊
SELECT session_type FROM b_session WHERE id = {sessionId}

-- 验证当前用户是否是群成员
SELECT COUNT(*) FROM b_session_user 
WHERE session_id = {sessionId} 
  AND user_id = {currentUserId}
  AND deleted_at IS NULL
```

**权限要求**:
- 会话必须是 `GROUP` 类型
- 当前用户必须是群成员（不限于群主，所有成员都可以邀请）

#### 2. 邀请成员
```sql
-- 对每个被邀请的用户
INSERT INTO b_session_user (
  session_id,
  user_id,
  session_user_type,
  created_at,
  updated_at
) VALUES (
  {sessionId},
  {invitedUserId},
  'MEMBER',
  NOW(),
  NOW()
)
ON DUPLICATE KEY UPDATE 
  deleted_at = NULL,  -- 如果之前退出过，重新加入
  updated_at = NOW()
```

#### 3. 发送通知
- 通过 WebSocket 向所有群成员推送"新成员加入"通知
- 向被邀请的用户推送"你被邀请加入群聊"通知

---

### 解散群聊接口

#### 1. 权限验证
```sql
-- 验证会话是否是群聊
SELECT session_type FROM b_session WHERE id = {sessionId}

-- 验证当前用户是否是群主
SELECT session_user_type FROM b_session_user 
WHERE session_id = {sessionId} 
  AND user_id = {currentUserId}
  AND deleted_at IS NULL
```

**权限要求**:
- 会话必须是 `GROUP` 类型
- 当前用户必须是 `OWNER`（群主）

#### 2. 解散群聊
```sql
-- 方案 1: 软删除（推荐）
-- 标记会话为已删除
UPDATE b_session 
SET deleted_at = NOW() 
WHERE id = {sessionId}

-- 标记所有成员关系为已删除
UPDATE b_session_user 
SET deleted_at = NOW() 
WHERE session_id = {sessionId}

-- 方案 2: 硬删除（不推荐，会丢失历史记录）
-- DELETE FROM b_session WHERE id = {sessionId}
-- DELETE FROM b_session_user WHERE session_id = {sessionId}
-- DELETE FROM b_message WHERE session_id = {sessionId}
```

**推荐使用软删除**，原因：
- 保留历史记录用于审计
- 可以实现"恢复群聊"功能
- 避免级联删除的复杂性

#### 3. 发送通知
- 通过 WebSocket 向所有群成员推送"群聊已解散"通知
- 前端收到通知后自动从会话列表移除该群聊

## 🔐 权限控制

### 邀请成员
- ✅ 所有群成员都可以邀请新成员
- ✅ 不能邀请已在群内的用户（自动跳过或返回提示）
- ✅ 被邀请的用户必须存在
- ❌ 非群成员无法邀请

### 解散群聊
- ✅ 只有群主可以解散群聊
- ✅ 解散后所有成员关系失效
- ✅ 解散后会话不再出现在任何人的会话列表中
- ❌ 普通成员无法解散

## 📝 错误码

### 邀请成员接口

| 状态码 | 错误信息 | 说明 |
|--------|---------|------|
| 200 | success | 邀请成功 |
| 400 | 该会话不是群聊 | session_type 不是 GROUP |
| 400 | 用户列表不能为空 | memberIds 为空 |
| 401 | 未认证 | Token 无效或过期 |
| 403 | 只有群成员可以邀请成员 | 当前用户不在群内 |
| 404 | 会话不存在 | sessionId 无效 |
| 404 | 用户不存在: {userId} | 某个被邀请的用户不存在 |

### 解散群聊接口

| 状态码 | 错误信息 | 说明 |
|--------|---------|------|
| 200 | success | 解散成功 |
| 400 | 该会话不是群聊 | session_type 不是 GROUP |
| 401 | 未认证 | Token 无效或过期 |
| 403 | 只有群主可以解散群聊 | 当前用户不是 OWNER |
| 404 | 会话不存在 | sessionId 无效 |

## 🎯 业务规则

### 邀请成员
1. **去重处理**: 如果用户已在群内，自动跳过（不报错）
2. **批量邀请**: 支持一次邀请多个用户
3. **失败处理**: 部分用户邀请失败不影响其他用户
4. **通知机制**: 邀请成功后立即通知所有群成员和被邀请者
5. **会话创建**: 被邀请的用户自动创建该群聊的 session_user 记录

### 解散群聊
1. **不可恢复**: 解散后群聊无法恢复（除非实现了恢复功能）
2. **全员通知**: 所有群成员都会收到解散通知
3. **会话清理**: 所有成员的会话列表中该群聊自动消失
4. **消息保留**: 建议保留消息记录用于审计（软删除）
5. **权限回收**: 解散后群主权限失效

---

## 📌 注意事项

### 邀请成员
1. **性能优化**: 批量邀请时使用批量插入，避免循环单条插入
2. **事务处理**: 邀请操作应该在事务中执行，保证原子性
3. **通知延迟**: WebSocket 通知可以异步发送，不阻塞接口响应
4. **用户验证**: 邀请前验证用户是否存在且是否已在群内

### 解散群聊
1. **二次确认**: 前端已实现二次确认，后端可以不做额外验证
2. **软删除优先**: 建议使用软删除，保留历史记录
3. **级联处理**: 解散时需要处理所有相关的 session_user 记录
4. **通知优先**: 先发送通知，再执行删除操作，避免通知失败
