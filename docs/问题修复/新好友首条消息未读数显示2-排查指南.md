# 新好友首条消息未读数显示 2 - 排查指南

## 🔍 问题描述

新加的好友发了第一条消息，接收方显示未读数为 2，但实际只有 1 条消息。

## 📊 可能的原因

### 原因 1: 后端返回的未读数就是 2

**排查方法**：
1. 打开浏览器开发者工具（F12）
2. 切换到 Network 标签
3. 刷新页面，查找 `/api/sessions` 或类似的会话列表请求
4. 查看响应数据中该会话的 `unreadCount` 字段

**如果后端返回 `unreadCount: 2`**：
- ✅ 这是后端问题
- 📝 参考：`docs/后端文档/未读数计算错误问题.md`
- 🔧 需要后端修复未读数计算逻辑

### 原因 2: 前端重复计数

**排查方法**：
1. 打开浏览器控制台（F12 -> Console）
2. 清空控制台
3. 等待接收消息
4. 查看以下日志：

```
📩 [receiveMessage] 处理接收到的消息: { id: 'xxx', ... }
⚠️ [receiveMessage] 消息已存在，跳过重复添加: xxx  ← 如果看到这个，说明消息重复了
✅ [receiveMessage] 消息已添加到 store
📊 [incrementUnread] 增加未读数
📊 [incrementUnread] 当前未读数: 1
📊 [incrementUnread] 新的未读数: 2
```

**如果看到多次 `incrementUnread`**：
- ✅ 这是前端问题
- 🔧 需要检查前端代码

### 原因 3: 会话创建时就有未读数

**排查方法**：
1. 在接收第一条消息前，查看控制台日志
2. 搜索 `loadSessions` 或 `会话列表`
3. 查看该会话的初始 `unreadCount`

```javascript
// 在控制台执行
const sessionStore = useSessionStore()
console.log('会话列表:', sessionStore.sessions.map(s => ({
  id: s.id,
  name: s.name,
  unreadCount: s.unreadCount
})))
```

**如果初始 `unreadCount` 就是 1**：
- ✅ 这是后端问题
- 📝 后端在创建会话时就设置了未读数
- 🔧 需要后端修复会话创建逻辑

## 🔬 详细排查步骤

### 步骤 1: 确认后端返回的数据

1. 打开开发者工具 Network 标签
2. 刷新页面
3. 找到会话列表请求（通常是 `/api/sessions` 或 `/api/session/list`）
4. 查看响应数据：

```json
{
  "sessions": [
    {
      "sessionId": "xxx",
      "name": "lyh2",
      "unreadCount": 1,  // ← 检查这个值
      "lastMessage": "在线",
      "lastMessageTime": "2024-02-27T10:00:00Z"
    }
  ]
}
```

**如果 `unreadCount` 是 2**：
- 问题在后端
- 跳转到"后端排查"部分

**如果 `unreadCount` 是 1**：
- 继续下一步

### 步骤 2: 监控 WebSocket 消息

1. 在控制台执行以下代码，监控 WebSocket 消息：

```javascript
// 保存原始的 WebSocket send 方法
const originalSend = WebSocket.prototype.send
WebSocket.prototype.send = function(...args) {
  console.log('📤 [WebSocket] 发送:', args[0])
  return originalSend.apply(this, args)

// 监控 WebSocket 消息
window.addEventListener('message', (event) => {
  if (event.data && event.data.type === 'websocket') {
    console.log('📥 [WebSocket] 接收:', event.data)
  }
})
```

2. 让好友发送一条消息
3. 查看控制台，确认是否收到重复的消息

在控制台执行：

## 🎯 后端排查

如果确认是后端问题，需要检查以下几点：

### 1. 会话创建时的未读数

**问题**：创建会话时就设置了 `unreadCount = 1`

**检查代码**：
```java
// 创建会话时
SessionUser sessionUser = new SessionUser();
sessionUser.setUnreadCount(0);  // ← 应该是 0，不是 1
```

### 2. 发送消息时的未读数更新

**问题**：发送消息时，给接收方的 `unreadCount` 增加了 2 次

**检查代码**：
```java
// 发送消息时
// 1. 保存消息
messageRepository.save(message);

// 2. 更新接收方的未读数（只应该执行一次）
sessionUserRepository.incrementUnreadCount(receiverSessionUser.getId());

// ❌ 错误：不要重复更新
// sessionUserRepository.incrementUnreadCount(receiverSessionUser.getId());
```

### 3. 会话列表查询

**问题**：查询会话列表时，未读数计算错误

**检查 SQL**：
```sql
-- 正确的查询
SELECT 
  s.session_id,
  s.name,
  su.unread_count,  -- ← 直接使用 session_user 表的 unread_count
  s.last_message
FROM sessions s
JOIN session_user su ON s.session_id = su.session_id
WHERE su.user_id = ?

-- ❌ 错误：不要重新计算未读数
-- SELECT 
--   s.session_id,
--   COUNT(m.id) as unread_count  -- ← 这样会导致计算错误
-- FROM sessions s
-- LEFT JOIN messages m ON m.session_id = s.session_id AND m.is_read = false
```

## 🔧 修复方案

### 前端修复（如果是前端问题）

已经实现了消息去重逻辑，理论上不应该出现重复计数。

如果仍然出现问题，可以添加额外的保护：

```typescript
// 在 websocket.ts 中
const receivedMessage = await messageStore.receiveMessage(message)

if (receivedMessage) {
  // 只有成功接收新消息时才增加未读数
  sessionStore.incrementUnread(message.sessionId)
} else {
  console.warn('⚠️ [handleIncomingMessage] 消息接收失败或重复，不增加未读数')
}
```

### 后端修复（如果是后端问题）

参考 `docs/后端文档/后端优化手册.md` 中的未读数计算优化。
