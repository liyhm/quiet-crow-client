# 消息重复发送和会话预览修复

## 📋 问题描述

### 问题 1: 群聊消息重复发送
- **现象**：发送一次消息，实际发送了 2 次
- **影响**：群聊和私聊都可能出现重复消息

### 问题 2: 会话列表最后一条消息显示错误
- **现象 1**：接收方显示 `{"field":"4f0ff80be..."}`（fileId）
- **现象 2**：发送方显示之前的旧消息，不是最新的消息
- **影响**：用户无法看到最新的消息预览

## 🔍 问题原因

### 问题 1: 消息重复的原因

**重复发送流程**：
```
1. 用户发送消息/文件
   ↓
2. 前端先本地添加消息（乐观更新）
   messageStore.addMessage(tempMessage)
   ↓
3. 通过 WebSocket 发送消息
   websocketService.sendMessage()
   ↓
4. WebSocket 回调收到自己发送的消息
   handleIncomingMessage()
   ↓
5. receiveMessage() 再次添加消息
   ❌ 导致重复！
```

**根本原因**：
- 乐观更新（先添加到本地）+ WebSocket 回调（再次添加）= 重复消息
- `receiveMessage` 虽然有去重逻辑，但临时 ID 和服务器返回的 ID 不同，无法去重

### 问题 2: 会话预览错误的原因

**原因 1：媒体消息预览格式问题**
- 媒体消息内容是 JSON：`{"fileId":"xxx","fileName":"xxx","fileSize":123}`
- 直接截取前 20 个字符：`{"field":"4f0ff80be...`
- 用户看到的是 JSON 字符串，而不是友好的预览

**原因 2：发送方未更新会话列表**
- 发送消息后，只更新了消息列表
- 没有调用 `sessionStore.updateLastMessage()`
- 导致会话列表显示旧消息

## ✅ 解决方案

### 修复 1: 移除乐观更新，避免重复

**修改文件**：`src/renderer/src/pages/main/MainPage.vue`

**修改前**：
```typescript
// 先在本地添加消息（乐观更新）
const tempMessage = {
  id: `temp-${Date.now()}`,
  sessionId: activeSessionId.value,
  senderId: currentUserId.value,
  content: messageContent,
  // ...
}

messageStore.addMessage(activeSessionId.value, tempMessage)

// 发送 WebSocket 消息
websocketService.sendMessage(activeSessionId.value, {
  sessionId: activeSessionId.value,
  content: encryptedContent,
  encrypted: true,
  messageType: messageType === 'IMAGE' ? '2' : '3'
})
```

**修改后**：
```typescript
// 发送 WebSocket 消息（不要先添加到本地，等 WebSocket 回调时再添加，避免重复）
websocketService.sendMessage(activeSessionId.value, {
  sessionId: activeSessionId.value,
  content: encryptedContent,
  encrypted: true,
  messageType: messageType === 'IMAGE' ? '2' : '3'
})

// 更新会话列表的最后一条消息预览
const preview = messageType === 'IMAGE' ? '[图片]' : `[文件] ${file.name}`
sessionStore.updateLastMessage(activeSessionId.value, preview, new Date().toISOString())

console.log('✅ 媒体消息已发送到 WebSocket')
```

**优点**：
- ✅ 避免重复消息
- ✅ 消息 ID 由服务器生成，保证唯一性
- ✅ 简化代码逻辑

**缺点**：
- ⚠️ 发送消息后，需要等待 WebSocket 回调才能看到消息（延迟约 100-300ms）
- ⚠️ 如果 WebSocket 断开，消息会丢失（但会加入队列，重连后发送）

### 修复 2: 优化媒体消息预览

**修改文件**：`src/renderer/src/shared/api/websocket.ts`

**修改前**：
```typescript
// 3. 更新会话的最后消息预览（使用解密后的内容）
const preview = receivedMessage.content?.substring(0, 20) || '新消息'
console.log('📨 [handleIncomingMessage] 更新会话预览:', preview)
```

**修改后**：
```typescript
// 3. 更新会话的最后消息预览（使用解密后的内容）
let preview = '新消息'

// 根据消息类型生成预览
if (receivedMessage.type === 'IMAGE') {
  preview = '[图片]'
} else if (receivedMessage.type === 'VIDEO') {
  preview = '[视频]'
} else if (receivedMessage.type === 'FILE') {
  // 尝试从 JSON 中提取文件名
  try {
    const fileData = JSON.parse(receivedMessage.content)
    preview = `[文件] ${fileData.fileName || '未知文件'}`
  } catch {
    preview = '[文件]'
  }
} else {
  // 文本消息
  preview = receivedMessage.content?.substring(0, 20) || '新消息'
}

console.log('📨 [handleIncomingMessage] 更新会话预览:', preview)
```

**效果**：
- ✅ 图片消息显示：`[图片]`
- ✅ 视频消息显示：`[视频]`
- ✅ 文件消息显示：`[文件] 文件名.pdf`
- ✅ 文本消息显示：前 20 个字符

### 修复 3: 发送方更新会话列表

**修改文件**：`src/renderer/src/entities/message/model/useMessageStore.ts`

**修改前**：

```typescript
const sendMessage = async (sessionId: string, content: string): Promise<void> => {
  // ... 发送逻辑 ...
  
  // 通过 WebSocket 发送
  websocketService.sendMessage(sessionId, {

```

**修改后**：
```typescript
const sendMessage = async (sessionId: string, content: string): Promise<void> => {
  // ... 发送逻辑 ...
  
  // 通过 WebSocket 发送
  websocketService.sendMessage(sessionId, {
    sessionId,
    content: encrypted,
    encrypted: true,
    messageType: '1'
  })

  // 更新会话列表的最后一条消息
  const { useSessionStore } = await import('@/entities/session/model/useSessionStore')
  const sessionStore = useSessionStore()
  sessionStore.updateLastMessage(sessionId, content.substring(0, 20), new Date().toISOString())

  // 模拟发送成功
  setTimeout(() => {
    updateMessageStatus(sessionId, message.id, MessageStatus.SENT)
  }, 500)
}
```

**效果**：
- ✅ 发送方的会话列表立即显示最新消息
- ✅ 与接收方保持一致

## ⚠️ 注意事项

### 1. 乐观更新的权衡

**移除乐观更新的影响**：
- ⚠️ 发送消息后，需要等待 WebSocket 回调才能看到消息（延迟约 100-300ms）
- ⚠️ 用户可能感觉消息发送有延迟

**如果需要保留乐观更新**：
- 需要实现更复杂的去重逻辑
- 临时 ID 需要与服务器返回的 ID 进行映射
- 需要处理发送失败的情况

### 2. WebSocket 断开时的消息处理

如果 WebSocket 断开：
- 消息会加入待发送队列
- 重连后自动发送
- 用户需要等待重连

### 3. 消息预览的长度

- 文本消息：前 20 个字符
- 媒体消息：固定格式（`[图片]`、`[视频]`、`[文件] 文件名`）
- 可以根据需要调整长度

### 4. 群聊名称的 JSON 格式

如果群聊名称显示 `{"field":"xxx"}`：
- 这是后端返回的 JSON 格式
- 需要在 `useSessionStore.ts` 的 `loadSessions` 中解析
- 参考之前的修复：解析 `{"field":"群名称"}` 格式
