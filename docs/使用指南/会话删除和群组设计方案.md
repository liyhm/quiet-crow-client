# 会话删除和群组设计方案

## 问题描述

当前实现：删除会话是软删除，只删除当前用户的 `session_user` 记录。

**问题**：
- 如果一方删除会话，对方还能看到
- 但如果要建群组，这个设计就有问题了

---

## 市面上聊天软件的解决方案

### 1. 微信的方案

#### 私聊
- **删除会话**：只删除自己的聊天记录，不影响对方
- **对方视角**：对方仍然可以看到会话和历史消息
- **重新发消息**：删除后，对方再发消息，会话会重新出现

#### 群聊
- **删除会话**：只删除自己的聊天记录，不影响其他成员
- **退出群聊**：退出后，不再接收新消息，但其他成员不受影响
- **解散群聊**：只有群主可以解散，解散后所有成员的群聊都消失

### 2. Telegram 的方案

#### 私聊
- **删除会话**：可以选择"只删除我的"或"双方都删除"
- **只删除我的**：只删除自己的记录
- **双方都删除**：双方的聊天记录都被删除（需要对方同意）

#### 群聊
- **删除会话**：只删除自己的记录
- **退出群聊**：退出后不再接收消息
- **删除群聊**：只有创建者可以删除

### 3. WhatsApp 的方案

#### 私聊
- **删除会话**：只删除自己的记录
- **删除消息**：可以选择"只删除我的"或"所有人都删除"（7分钟内）

#### 群聊
- **删除会话**：只删除自己的记录
- **退出群聊**：退出后不再接收消息
- **删除群聊**：只有管理员可以删除

---

## 推荐方案

### 方案 A：微信模式（推荐）

**设计原则**：
- 删除会话 = 删除自己的聊天记录
- 不影响其他人
- 简单、清晰、用户容易理解

#### 数据库设计

**表结构**：
```
b_session (会话表)
- id: 会话ID
- type: 会话类型（PRIVATE/GROUP）
- created_time: 创建时间

b_session_user (会话成员表)
- id: 记录ID
- session_id: 会话ID
- user_id: 用户ID
- role: 角色（OWNER/ADMIN/MEMBER）
- joined_time: 加入时间
- deleted_time: 删除时间（软删除）
- is_deleted: 是否已删除

b_session_log (消息表)
- id: 消息ID
- session_id: 会话ID
- send_user: 发送者ID
- message_content: 消息内容
- send_time: 发送时间
```

#### 删除会话逻辑

**私聊**：
1. 用户 A 删除会话
2. 更新 `b_session_user` 表：
   - 设置 `is_deleted = true`
   - 设置 `deleted_time = 当前时间`
3. 用户 A 的会话列表中不再显示该会话
4. 用户 B 不受影响，仍然可以看到会话

**重新出现**：
- 用户 B 发送新消息
- 用户 A 收到消息
- 检测到 `is_deleted = true`，自动恢复：
  - 设置 `is_deleted = false`
  - 清空 `deleted_time`
- 会话重新出现在用户 A 的列表中

**群聊**：
1. 用户删除会话：只删除自己的记录
2. 用户退出群聊：
   - 设置 `is_deleted = true`
   - 不再接收新消息
3. 群主解散群聊：
   - 删除 `b_session` 记录
   - 所有成员的会话都消失

---

### 方案 B：Telegram 模式（更灵活）

**设计原则**：
- 提供更多选项
- 用户可以选择删除方式

#### 删除选项

**私聊**：
- **只删除我的**：只删除自己的记录
- **双方都删除**：发送删除请求，对方同意后双方都删除

**群聊**：
- **删除会话**：只删除自己的记录
- **退出群聊**：退出并删除记录

#### 实现复杂度

**优点**：
- 更灵活
- 用户有更多控制权

**缺点**：
- 实现复杂
- 需要额外的确认流程
- 用户可能不理解

### 为什么选择方案 A？

1. **简单易懂**：用户容易理解
2. **实现简单**：不需要复杂的确认流程
3. **符合习惯**：大多数用户习惯微信的方式

## 具体实现

### 1. 数据库修改

**b_session_user 表**：
```sql
ALTER TABLE b_session_user 
ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE,
ADD COLUMN deleted_time DATETIME NULL;
```

### 2. 删除会话 API

**接口**：`DELETE /api/session/{sessionId}`

**逻辑**：
1. 查找当前用户的 `session_user` 记录
2. 设置 `is_deleted = true`
3. 设置 `deleted_time = 当前时间`
4. 不删除 `session` 和 `message` 记录

**SQL**：
```sql
UPDATE b_session_user 
SET is_deleted = true, deleted_time = NOW()
WHERE session_id = ? AND user_id = ?
```

### 3. 获取会话列表 API

**接口**：`GET /api/session/my`

**逻辑**：
1. 查询当前用户的会话
2. 过滤掉 `is_deleted = true` 的记录

**SQL**：
```sql
SELECT * FROM b_session_user 
WHERE user_id = ? AND is_deleted = false
ORDER BY updated_time DESC
```

### 4. 接收消息时自动恢复

**逻辑**：
1. 用户收到新消息
2. 检查 `session_user` 记录
3. 如果 `is_deleted = true`，自动恢复：
   ```sql
   UPDATE b_session_user 
   SET is_deleted = false, deleted_time = NULL
   WHERE session_id = ? AND user_id = ?
   ```
4. 会话重新出现在列表中

## 群聊设计

### 群聊类型

**普通群**：
- 任何成员都可以邀请新成员
- 只有群主可以解散群聊

**私密群**：
- 只有管理员可以邀请新成员
- 只有群主可以解散群聊

### 群聊角色

**OWNER（群主）**：
- 可以解散群聊
- 可以添加/移除管理员
- 可以添加/移除成员
- 可以修改群信息

**ADMIN（管理员）**：
- 可以添加/移除成员
- 可以修改群信息

**MEMBER（普通成员）**：
- 可以发送消息
- 可以退出群聊

### 群聊操作

#### 1. 创建群聊

**接口**：`POST /api/session/group`

**请求**：
```json
{
  "name": "我的群聊",
  "memberIds": ["user1", "user2", "user3"]
}
```

**逻辑**：
1. 创建 `session` 记录（type = GROUP）
2. 创建群主的 `session_user` 记录（role = OWNER）
3. 创建成员的 `session_user` 记录（role = MEMBER）
4. 生成群聊密钥

#### 2. 退出群聊

**接口**：`POST /api/session/{sessionId}/leave`

**逻辑**：
1. 设置 `is_deleted = true`
2. 不再接收新消息
3. 其他成员不受影响

#### 3. 解散群聊

**接口**：`DELETE /api/session/{sessionId}/disband`

**权限**：只有群主可以操作

**逻辑**：
1. 删除 `session` 记录
2. 所有成员的 `session_user` 记录设置为 `is_deleted = true`
3. 所有成员的会话列表中不再显示该群聊

#### 4. 添加成员

**接口**：`POST /api/session/{sessionId}/members`

**请求**：
```json
{
  "userIds": ["user4", "user5"]
}
```

**逻辑**：
1. 验证权限（群主或管理员）
2. 创建新成员的 `session_user` 记录
3. 通知新成员加入群聊

#### 5. 移除成员

**接口**：`DELETE /api/session/{sessionId}/members/{userId}`

**权限**：群主或管理员

**逻辑**：
1. 设置该成员的 `is_deleted = true`
2. 该成员不再接收新消息

## 前端实现

### 1. 删除会话

**当前实现**：
```typescript
const deleteSession = async (sessionId: string) => {
  await sessionApi.deleteSession(sessionId)
  sessions.value = sessions.value.filter(s => s.id !== sessionId)
}
```

**保持不变**：前端逻辑不需要修改

### 2. 接收消息时恢复会话

**在 WebSocket 消息处理中**：
```typescript
private async handleIncomingMessage(message: any) {
  // 检查会话是否存在
  const sessionExists = sessionStore.sessions.find(s => s.id === message.sessionId)
  
  if (!sessionExists) {
    // 会话不存在，重新加载会话列表
    // 后端会自动恢复 is_deleted = false
    await sessionStore.loadSessions()
  }
  
  // 处理消息...
}
```

**已实现**：前端已经有这个逻辑

## 测试场景

### 场景 1：私聊删除会话

1. 用户 A 和用户 B 聊天
2. 用户 A 删除会话
3. 用户 A 的会话列表中不再显示
4. 用户 B 的会话列表中仍然显示
5. 用户 B 发送新消息
6. 用户 A 收到消息，会话重新出现

### 场景 2：群聊退出

1. 用户 A、B、C 在一个群聊中
2. 用户 A 退出群聊
3. 用户 A 的会话列表中不再显示该群聊
4. 用户 B、C 仍然可以在群聊中发消息
5. 用户 A 不再接收新消息

### 场景 3：群聊解散

1. 用户 A（群主）、B、C 在一个群聊中
2. 用户 A 解散群聊
3. 所有成员的会话列表中不再显示该群聊
4. 无法再发送消息

## 总结

### 推荐方案：微信模式

**私聊**：
- 删除会话 = 软删除（只删除自己的记录）
- 对方发消息后，会话自动恢复

**群聊**：
- 删除会话 = 软删除（只删除自己的记录）
- 退出群聊 = 不再接收消息
- 解散群聊 = 所有成员的会话都消失（只有群主可以）

### 优点

1. **简单易懂**：用户容易理解
2. **实现简单**：不需要复杂的确认流程
3. **符合习惯**：大多数用户习惯这种方式
4. **适合群聊**：群聊场景下更合理
