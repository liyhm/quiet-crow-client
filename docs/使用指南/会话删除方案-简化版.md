# 会话删除方案 - 简化版

## 核心原则

1. **删除就是真删除**：删除 `session_user` 记录
2. **不自动恢复**：删除后不会自动恢复
3. **收到新消息时重新创建**：对方发消息后，会话重新出现

---

## 设计方案

### 场景 1：用户 A 删除会话

**操作**：
1. 用户 A 删除与用户 B 的会话
2. 删除 `b_session_user` 表中用户 A 的记录
3. 用户 A 的会话列表中不再显示该会话

**数据库操作**：
```sql
DELETE FROM b_session_user 
WHERE session_id = ? AND user_id = ?
```

**结果**：
- 用户 A：会话消失
- 用户 B：会话仍然存在，不受影响

---

### 场景 2：用户 B 发送新消息

**操作**：
1. 用户 B 发送消息给用户 A
2. 后端检测到用户 A 没有 `session_user` 记录
3. 自动为用户 A 创建新的 `session_user` 记录
4. 用户 A 收到消息，会话重新出现

**后端逻辑**：
```
1. 用户 B 发送消息
2. 查询会话的所有成员
3. 发现用户 A 没有 session_user 记录
4. 自动创建用户 A 的 session_user 记录
5. 转发消息给用户 A
6. 用户 A 收到消息，会话重新出现
```

**数据库操作**：
```sql
-- 1. 检查用户 A 是否有 session_user 记录
SELECT * FROM b_session_user 
WHERE session_id = ? AND user_id = ?

-- 2. 如果不存在，创建新记录
INSERT INTO b_session_user (id, session_id, user_id, session_user_type, created_time)
VALUES (?, ?, ?, 'PRIVATE', NOW())
```

## 后端需求

### 需求 1：删除会话（真删除）

**接口**：`DELETE /api/session/{sessionId}`

**逻辑**：
1. 删除当前用户的 `session_user` 记录（真删除，不是软删除）
2. 不删除 `session` 表记录
3. 不删除 `message` 表记录
4. 不影响其他用户

**SQL**：
```sql
DELETE FROM b_session_user 
WHERE session_id = ? AND user_id = ?
```

**响应**：
```json
{
  "code": 200,
  "message": "会话已删除",
  "data": "会话已删除"
}
```

---

### 需求 2：发送消息时自动创建 session_user

**触发时机**：用户发送消息时

**逻辑**：
1. 获取会话的所有应该存在的成员（从会话类型判断）
2. 检查每个成员是否有 `session_user` 记录
3. 如果没有，自动创建

**实现位置**：消息转发逻辑

**伪代码**：
```java
public void forwardMessage(String sessionId, Message message) {
    // 1. 获取会话信息
    Session session = sessionRepository.findById(sessionId);
    
    // 2. 获取应该存在的成员
    List<String> expectedMembers = getExpectedMembers(session);
    
    // 3. 检查并创建缺失的 session_user 记录
    for (String userId : expectedMembers) {
        ensureSessionUserExists(sessionId, userId, session.getType());
    }
    
    // 4. 转发消息
    for (String userId : expectedMembers) {
        if (!userId.equals(message.getSendUser())) {
            messagingTemplate.convertAndSendToUser(userId, "/queue/messages", message);
        }
    }
}

private void ensureSessionUserExists(String sessionId, String userId, String sessionType) {
    // 检查是否存在
    boolean exists = sessionUserRepository.exists(sessionId, userId);
    
    if (!exists) {
        // 自动创建
        SessionUser sessionUser = new SessionUser();
        sessionUser.setSessionId(sessionId);
        sessionUser.setUserId(userId);
        sessionUser.setSessionUserType(sessionType);
        sessionUser.setCreatedTime(LocalDateTime.now());
        sessionUserRepository.save(sessionUser);
        
        log.info("自动创建 session_user 记录: sessionId={}, userId={}", sessionId, userId);
    }
}

private List<String> getExpectedMembers(Session session) {
    if ("PRIVATE".equals(session.getType())) {
        // 私聊：从 session 表的 participants 字段获取
        // 或者从第一次创建时保存的成员列表获取
        return getPrivateChatMembers(session.getId());
    } else {
        // 群聊：从现有的 session_user 记录获取
        return sessionUserRepository.findUserIdsBySessionId(session.getId());
    }
}
```

### 需求 3：私聊会话的成员管理

**问题**：删除 `session_user` 后，如何知道私聊的另一方是谁？

**方案 A：在 session 表保存成员列表**

**修改 b_session 表**：
```sql
ALTER TABLE b_session 
ADD COLUMN participants TEXT COMMENT '成员列表（JSON格式）';
```

**示例数据**：
```json
{
  "participants": ["user-a-id", "user-b-id"]
}
```

**优点**：
- 简单直接
- 删除 session_user 后仍然知道成员

**缺点**：
- 需要修改表结构

---

**方案 B：从消息记录推断**

**逻辑**：
```sql
-- 从消息记录中获取参与者
SELECT DISTINCT send_user 
FROM b_session_log 
WHERE session_id = ?
```

**优点**：
- 不需要修改表结构

**缺点**：
- 如果没有消息记录，无法推断
- 性能稍差

---

**推荐方案 A**：在 `session` 表保存成员列表

## 群聊场景

### 群聊删除会话

**操作**：
1. 用户 A 删除群聊会话
2. 删除用户 A 的 `session_user` 记录
3. 用户 A 不再接收群消息

**问题**：如果有人在群里发消息，用户 A 会重新收到吗？

**方案 1：不自动重新加入**
- 删除后，即使有人发消息，也不自动创建 `session_user`
- 用户 A 不再接收群消息
- 除非用户 A 主动重新加入群聊

**方案 2：自动重新加入**
- 删除后，如果有人发消息，自动创建 `session_user`
- 用户 A 重新接收群消息

**推荐**：方案 1（不自动重新加入）

**实现**：
```java
private void ensureSessionUserExists(String sessionId, String userId, String sessionType) {
    // 只对私聊自动创建
    if ("PRIVATE".equals(sessionType)) {
        boolean exists = sessionUserRepository.exists(sessionId, userId);
        if (!exists) {
            // 自动创建
            createSessionUser(sessionId, userId, sessionType);
        }
    }
    // 群聊不自动创建
}
```

## 总结

### 核心逻辑

1. **删除会话**：真删除 `session_user` 记录
2. **私聊**：对方发消息后，自动创建 `session_user`，会话重新出现
3. **群聊**：删除后不自动重新加入，除非用户主动加入

### 数据库修改

**b_session 表**：
```sql
ALTER TABLE b_session 
ADD COLUMN participants TEXT COMMENT '成员列表（JSON格式，私聊时使用）';
```

### 后端需要实现

1. 删除会话 API（真删除）
2. 发送消息时自动创建 `session_user`（仅私聊）
3. 创建会话时保存 `participants` 字段

### 前端无需修改

- 删除会话逻辑已正确
- 接收消息时自动加载会话列表已正确

## 用户体验

### 私聊

**场景**：
1. 用户 A 删除与用户 B 的会话
2. 用户 A 的会话列表中不再显示
3. 用户 B 发送消息："你好"
4. 用户 A 收到消息，会话重新出现
5. 用户 A 可以看到新消息

**优点**：
- 简单直接
- 符合用户预期
- 不会丢失消息

### 群聊

**场景**：
1. 用户 A 删除群聊会话
2. 用户 A 的会话列表中不再显示
3. 其他人在群里发消息
4. 用户 A 不会收到消息（因为没有 `session_user` 记录）
5. 用户 A 需要主动重新加入群聊

**优点**：
- 删除后真的不再接收消息
- 避免骚扰

**缺点**：
- 可能会错过重要消息

**改进**：
- 提供"退出群聊"和"删除会话"两个选项
- "删除会话"：只删除本地记录，仍然接收消息
- "退出群聊"：真正退出，不再接收消息
