### 1. API 端点配置更新

```typescript
SESSION: {

  READ_BATCH: (sessionId: string) => `/api/session/${sessionId}/read/batch`,     // 批量标记

```

### 2. Session API 更新

在 `src/renderer/src/shared/api/session.ts` 中添加了三个方法：

```typescript
// 1. 标记单条消息已读（推荐）
markSingleAsRead(messageId: string)

// 2. 标记到某条消息（批量标记该消息及之前的所有消息）
markAsRead(sessionId: string, messageId: string)

// 3. 批量标记多条消息已读
markBatchAsRead(sessionId: string, messageIds: string[])
```

### 3. Message Store 更新

在 `src/renderer/src/entities/message/model/useMessageStore.ts` 中：

- `markAsRead()` - 改用推荐的单条消息标记接口
- `markBatchAsRead()` - 新增批量标记功能

### 4. 主页面逻辑优化

在 `src/renderer/src/pages/main/MainPage.vue` 中：

**优化前**：
- 只标记最后一条消息
- 不区分是否是自己发送的

**优化后**：
- 筛选出对方发送的未读消息
- 批量标记所有未读消息
- 避免标记自己发送的消息

```typescript
const selectSession = async (sessionId: string): Promise<void> => {
  // 加载消息
  await messageStore.loadMessages(sessionId)
  
  // 筛选对方发送的未读消息
  const unreadMessages = messages.filter(m => 
    !m.isRead && m.senderId !== authStore.userId
  )
  
  // 批量标记已读
  if (unreadMessages.length > 0) {
    await messageStore.markBatchAsRead(sessionId, unreadIds)
  }
}
```

## 当前问题排查

### 问题：每次刷新都显示未读

**可能原因**：
1. 后端没有正确保存已读状态
2. 前端标记已读的调用失败
3. 会话列表的未读数统计有问题

**排查步骤**：

1. **检查标记已读是否成功**

打开控制台，选择一个会话，查看是否有：
```
📖 标记 X 条未读消息为已读
✅ 批量标记已读成功，共 X 条消息
```

如果没有这些日志，说明标记失败。

2. **检查后端响应**

在 Network 标签中查看：
```
POST /api/session/{sessionId}/read/batch
Status: 200 OK
```

如果返回错误，检查后端日志。

3. **检查数据库**

```sql
-- 查看消息的已读状态
SELECT id, send_user, is_read 
FROM b_session_log 
WHERE session_id = 'xxx' 
ORDER BY send_time DESC;
```

如果 `is_read` 仍然是 `false`，说明后端没有更新。

4. **检查会话列表的未读数**

```sql
-- 统计未读消息数
SELECT COUNT(*) 
FROM b_session_log 
WHERE session_id = 'xxx' 
  AND is_read = false 
  AND send_user != 'current-user-id';
```

## 消息显示问题排查

### 问题：所有消息都在左边

**原因**：`senderId` 和 `currentUserId` 不匹配

**排查**：

在控制台查看：
```
📨 当前消息列表: {
  currentUserId: "xxx",
  messages: [
    { senderId: "yyy", isMine: false, content: "..." }
  ]
}
```

**关键检查点**：
1. `currentUserId` 是否正确（应该是当前登录用户的 ID）
2. `senderId` 是否正确（应该是发送者的 ID）
3. `isMine` 是否正确计算（`senderId === currentUserId`）

**如果 `senderId` 和 `currentUserId` 格式不同**：
- 可能一个是 UUID，一个是用户名
- 需要统一使用 UUID 或用户名

### 问题：消息内容不显示

**可能原因**：
1. 解密失败
2. `content` 字段为空
3. 后端返回的数据有问题

**排查**：

查看控制台：
```
🔓 解密消息: { id: xxx, sendUser: xxx, encrypted: xxx }
✅ 解密成功: 你好
```

如果看到 `❌ 解密消息失败`，说明：
- 会话密钥不正确
- 加密算法不匹配
- 消息内容损坏
