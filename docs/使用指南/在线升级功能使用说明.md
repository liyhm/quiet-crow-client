# 在线升级功能使用说明

## 📦 功能概述

鸦口无言现在支持在线自动更新功能，用户可以在应用内直接检查、下载和安装新版本，无需手动下载安装包。

## ✨ 功能特点

- 🔍 自动检查更新（启动后 5 秒自动检查）
- 📥 后台下载更新包
- 📊 实时显示下载进度
- 🚀 一键安装重启
- 🎨 赛博朋克风格 UI
- 🔒 安全可靠（使用 electron-updater）

## 🎯 使用方法

### 方法 1: 手动检查更新

1. 点击左下角的"设置"按钮（齿轮图标）
2. 在弹出的设置面板中，点击"检查更新"
3. 系统会自动检查是否有新版本

### 方法 2: 自动检查

应用启动后 5 秒会自动在后台检查更新，如果有新版本会弹出提示。

## 📱 更新流程

### 1. 检查更新

```
应用启动
  ↓
等待 5 秒
  ↓
后台检查更新
  ↓
发现新版本 → 显示更新弹窗
没有新版本 → 静默（不打扰用户）
```

### 2. 下载更新

```
点击"立即更新"
  ↓
后台下载更新包
  ↓
显示下载进度（百分比 + 文件大小）
  ↓
下载完成 → 提示"准备安装"
```

### 3. 安装更新

```
点击"立即重启"
  ↓
应用自动关闭
  ↓
安装新版本
  ↓
自动重启应用
```

## 🖼️ 界面说明

### 更新弹窗状态

1. **检查中**
   - 显示加载动画
   - 提示"正在检查更新..."

2. **发现新版本**
   - 显示当前版本和最新版本对比
   - 显示更新内容（如果有）
   - 按钮：关闭 / 立即更新

3. **下载中**
   - 显示进度条
   - 显示百分比和文件大小
   - 无法关闭（防止中断下载）

4. **下载完成**
   - 显示成功图标
   - 提示"更新已下载完成"
   - 按钮：稍后重启 / 立即重启

5. **已是最新**
   - 显示成功图标
   - 显示当前版本号
   - 按钮：关闭 / 重新检查

6. **更新失败**
   - 显示错误图标
   - 显示错误信息
   - 按钮：关闭 / 重新检查

## ⚙️ 配置说明

### 开发环境

开发环境下不会检查更新，避免干扰开发。

### 生产环境

需要配置更新服务器地址。

#### 修改配置

编辑 `src/main/index.ts`，找到以下代码：

```typescript
// 配置自动更新
function setupAutoUpdater(): void {
  // 设置更新服务器地址（需要替换为你的服务器地址）
  autoUpdater.setFeedURL({
    provider: 'generic',
    url: 'https://your-server.com/updates'
  })
  
  // ... 其他配置
}
```

将 `https://your-server.com/updates` 替换为你的更新服务器地址。

## 🌐 更新服务器搭建

### 方案 1: GitHub Releases（推荐）

使用 GitHub Releases 作为更新服务器，完全免费。

#### 配置步骤

1. 修改 `electron-builder.yml`：

```yaml
publish:
  provider: github
  owner: your-username
  repo: your-repo-name
```

2. 修改 `src/main/index.ts`：

```typescript
autoUpdater.setFeedURL({
  provider: 'github',
  owner: 'your-username',
  repo: 'your-repo-name'
})
```

3. 打包并发布：

```bash
# 打包
npm run build:win

# 发布到 GitHub Releases
# 需要设置 GH_TOKEN 环境变量
set GH_TOKEN=your_github_token
npm run build:win -- --publish always
```

### 方案 2: 自建服务器

#### 服务器要求

- 支持 HTTPS
- 可以托管静态文件
- 需要以下文件结构：

```
/updates/
  ├── latest.yml          # 版本信息文件
  ├── your-app-1.0.0.exe  # 安装包
  └── your-app-1.0.1.exe  # 新版本安装包
```

#### latest.yml 格式

```yaml
version: 1.0.1
releaseDate: '2024-02-27T10:00:00.000Z'
files:
  - url: your-app-1.0.1.exe
    sha512: <文件的 SHA512 哈希值>
    size: 123456789
path: your-app-1.0.1.exe
sha512: <文件的 SHA512 哈希值>
releaseNotes: |
  ## 更新内容
  - 新增在线升级功能
  - 修复已知问题
  - 性能优化
```

#### 配置更新地址

```typescript
autoUpdater.setFeedURL({
  provider: 'generic',
  url: 'https://your-server.com/updates'
})
```

### 方案 3: 云存储（OSS/S3）

使用阿里云 OSS、腾讯云 COS 或 AWS S3 等云存储服务。

#### 配置示例（阿里云 OSS）

```typescript
autoUpdater.setFeedURL({
  provider: 'generic',
  url: 'https://your-bucket.oss-cn-hangzhou.aliyuncs.com/updates'
})
```

## 🔧 打包配置

### electron-builder.yml

```yaml

productName: 鸦口无言
directories:
  output: dist
  buildResources: resources

# 自动更新配置

  provider: generic
  url: https://your-server.com/updates

  target:
    - target: nsis
      arch:
        - x64

# NSIS 安装程序配置

  oneClick: false
  allowToChangeInstallationDirectory: true
  createDesktopShortcut: true
  createStartMenuShortcut: true
  shortcutName: 鸦口无言
```

## 📝 版本号管理

### package.json

```json
{
  "name": "quiet",

  ...
}
```

每次发布新版本时，需要更新 `version` 字段：

```bash
# 小版本更新（bug 修复）

npm version minor  # 1.0.0 -> 1.1.0

# 大版本更新（重大变更）
npm version major  # 1.0.0 -> 2.0.0
```

## ⚠️ 注意事项

### 1. 开发环境

开发环境下不会检查更新，避免干扰开发。

### 2. 版本号格式

必须使用语义化版本号（Semantic Versioning）：

- ✅ 正确：`1.0.0`, `1.2.3`, `2.0.0-beta.1`
- ❌ 错误：`v1.0`, `1.0`, `version-1`

### 3. 文件完整性

更新包必须包含正确的 SHA512 哈希值，否则会验证失败。

### 4. HTTPS 要求

更新服务器必须使用 HTTPS，否则会被浏览器阻止。

### 5. 防火墙

确保应用可以访问更新服务器，某些企业防火墙可能会阻止。

### 6. 磁盘空间

下载更新包需要足够的磁盘空间（通常是安装包大小的 2 倍）。

## 🐛 常见问题

### Q1: 检查更新失败

**原因**：
- 更新服务器地址错误
- 网络连接问题
- 服务器未配置 CORS

**解决方法**：
1. 检查更新服务器地址是否正确
2. 检查网络连接
3. 查看控制台错误信息

### Q2: 下载失败

**原因**：
- 网络中断
- 磁盘空间不足
- 文件损坏

**解决方法**：
1. 检查网络连接
2. 检查磁盘空间
3. 重新下载

### Q3: 安装失败

**原因**：
- 权限不足
- 应用正在运行
- 文件被占用

**解决方法**：
1. 以管理员身份运行
2. 关闭所有应用实例
3. 重启电脑后重试

### Q4: 更新后版本号没变

**原因**：
- 缓存问题
- 安装失败

**解决方法**：
1. 完全卸载旧版本
2. 重新安装新版本
3. 清除应用数据

## 🔐 安全建议

1. **使用 HTTPS**：更新服务器必须使用 HTTPS
2. **验证签名**：验证更新包的数字签名
3. **哈希校验**：验证文件的 SHA512 哈希值
4. **增量更新**：大文件可以使用增量更新减少下载量

1. **定期发布**：建议每 1-2 周发布一次更新
2. **版本说明**：每次更新都要写清楚更新内容
3. **灰度发布**：重大更新可以先发布给部分用户测试
4. **回滚机制**：保留旧版本，出问题可以快速回滚
5. **用户通知**：重要更新要提前通知用户
